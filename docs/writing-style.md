# Writing Style

Prefer short sentences.

One idea per sentence.

One idea per paragraph whenever practical.

Avoid repeating concepts.

Reference instead of repeating.

---

Headings

Headings should describe.

Not persuade.

---

Lists

Use lists only when they improve readability.

Do not convert every paragraph into bullets.

---

Examples

Examples explain.

They do not define the framework.

# Editorial Principles

Every document should have a single responsibility.

Every concept should have a single source of truth.

Normative documents define.

Informative documents explain.

Examples illustrate.

Reflection statements inspire.

Do not duplicate concepts across documents.

Reference instead of repeating.

# Voice

Ship It! speaks plainly.

It avoids hype.

It avoids buzzwords.

It avoids exaggeration.

---

Prefer

Simple.

Clear.

Calm.

Direct.

---

Avoid

Revolutionary.

Game-changing.

Next generation.

AI-powered.

World-class.

Enterprise-ready.

---

The framework should sound confident without sounding certain.

It should invite discussion rather than end it.

# Recognition Test

Before adding any new sentence, ask:

Can an experienced software developer honestly say:

"I've seen this happen."

If not,

rewrite it.

---

Describe observations.

Not marketing.

---

Describe reality.

Not aspirations.

---

Avoid attacking other frameworks.

Instead, describe the problem Ship It! simplifies.

---

The reader should recognize the problem

before understanding the solution.
