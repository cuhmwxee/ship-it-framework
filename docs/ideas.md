# The Problem

Modern software produces small changes continuously.

Many delivery processes were designed around a single model for every change.

The problem appears when small changes travel through processes designed for something larger.

Not every change needs the same process.

Ship It! helps teams rethink what is actually required before shipping.

## Notes

The framework is not anti-process.

The question is whether the process still serves software delivery.

Examples:

- Dependency updates
- Security patches
- Configuration changes
- AI-generated changes
- Small bug fixes

Developer thoughts:

- "Why is this so difficult?"
- "Shouldn't this already be shipped?"
- "What are we still waiting for?"
- "Who still needs to validate this?"
- "Why does this small change need such a large process?"
- "Am I improving the software or feeding the process?"

Open questions:

- What is actually required before shipping?
- Which parts of the process create confidence?
- Which parts survived after their original purpose disappeared?
- When did process become an end in itself?

What is Ship It?

Ship It! is a minimal software delivery framework.

Why does it exist?
